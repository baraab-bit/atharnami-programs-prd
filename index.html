Product Requirements · v2 · Draft for approval

Programs that repeat, measured per run.

Editions, Assessments & the Input → Indicator Link.

Atharnami's core unit today is a single flat Program with a single flat Theory of Change — and it breaks the moment reality arrives. This PRD introduces a clean hierarchy — Organization → Program → Edition → Assessment — where the Assessment owns the ToC, plus a source-agnostic Link that turns a survey question into a reviewed indicator value with full provenance. Along the way it removes two concepts that added confusion without truth.

For review by  Anas Owner  Baraa · Qararat Drafted & revised  2026-06-30 Author  Nami
§1 — One paragraph

The whole thing, in one breath.

A real program repeats — every year, or several cohorts a year. It is evaluated per run, sometimes pre and post, sometimes for different groups of people — and each of those needs its own ToC and its own numbers. On top of that, even when data is collected, there is no way to turn a survey into an indicator's number — the pipeline was never built.

This PRD introduces a clean hierarchy where the Assessment owns the ToC, and a source-agnostic Link that turns a survey question — later interviews, records — into a reviewed indicator value with full provenance.
What's new

A hierarchy that holds reality

Organization → Program → Edition (a run / cohort) → Assessment (Pre/Post × group). The Assessment owns its ToC, survey, values and snapshots.

The heart

An indicator's number is a sentence

A source-agnostic Link: source → selector → aggregation → reviewed number, with live preview, drill-back and provenance.

The subtraction

Two concepts removed

Partners and Funders/Grants — both added confusion without truth. The Organization becomes the top of the hierarchy.

§2 — From a real program

Two walls, opening one program.

The consultant opens الاستدامة المالية (Financial Sustainability) under Salem bin Mahfouz. The post data is collected and imported. Then the model fails.

Wall 1 — Structure

The ToC can't hold a year, a cohort, or a pre/post.

It's one flat map per program — no way to go back to last year, or to hold the Pre and the Post side by side.

Wall 2 — Pipeline

There's nowhere to put the survey.

linked_survey_id points at a whole survey and nothing reads it. There is no rule for how N responses become one number, and no place the number lands.

The live data proved it: this one real program exists as two disconnected records — …للمؤسسات and …٢٠٢٥ بعدي — because there was no Edition / Assessment layer to hold them together. The model below merges them: 2025 Post and 2026 Pre of one Program.
§3 — The model

One hierarchy, parent to leaf.

The Assessment is the centre of gravity: it owns the ToC, the survey, the values and the snapshots. Everything above it is identity; everything below it is measurement.

Load-bearing decisions (resolved)

01 · Independence

Editions are independent and frozen. Editing 2025 never propagates to 2026.

02 · Ownership

The ToC belongs to the Assessment. Pre/Post and different groups can have genuinely different ToCs — so pre↔post, group, and year-over-year are all the same clone-and-match relationship.

03 · Strictness

Assessment type is strictly Pre / Post. A different group is a separate Assessment, not a third type.

04 · Direction

Comparison runs along a group, through time — Students Pre→Post, Students 2025→2026. Never across groups. Optional, lineage-driven, Phase 2.

05 · The common case stays trivial

A post-only, single-group program = one Edition, one Assessment, one ToC, one number per indicator. The breadcrumb is always shown — الاستدامة المالية › 2025 › Post.

§4 — The Input→Indicator Link

The heart: an indicator's number is a sentence.

The consultant never "writes a metric." They point at a question and choose how to count it. Each clause is an editable pill.

v1 = one indicator ← one binding (one question). ratio covers two inputs; custom_formula is the escape hatch. A dedicated multi-question index/scale builder is deferred to a later phase.

The "Compute & review" beat — why it's trusted

On the indicator, "Link a source" → Survey → this assessment's survey → question.
Pick aggregation (UI pre-suggests by field type; percent_selecting reveals "which options count as yes").
Live preview: the number + n= appears before commit — SAR 412,000 (n=38).
"Drill" → see the actual responses behind it. Lineage from day one.
"Compute & review" → number lands in this assessment, stamped source = measured, contributing rows snapshotted.
New responses later? The trusted number does not silently move — the indicator badges "Stale: 12 new responses, review again." Manual default; auto-refresh optional.
Source-agnostic by design

Interviews & records next

A Link is source_type (survey · interview · record · api · manual) + selector + aggregation + provenance. An interview later is just source_type=interview, where the "aggregation" may be an AI-suggested code you validate — same object, same review beat, same drill-back.

Authoring entry points & scope

From the indicator, primarily

Primary — "Link a source" on the ToC node: the universal gesture, the only one that works for records / interviews. Accelerator — from the survey, pre-tag a question at build time. The picker shows only this program · this edition · this assessment's surveys. One survey → one edition → one assessment.

§5 — Navigation & the Program Overview

You pick what to enter. No guessed scope, ever.

Clicking a program opens a Program Overview — a new page listing the program's Editions (grouped by year), each Edition's Assessments (Pre/Post × group), with status and history.

The landing

Program Overview

You enter the working ToC view only after choosing an edition + assessment; the always-on breadcrumb + scope bar carries it from there. Old ?program= links land here safely. The Overview is also the future home of the Phase-2 year-over-year comparison — the "history" is already its spine.

Health score — formula

Per-assessment target attainment

For each indicator with a target — higher-is-better min(actual÷target,1)×100, lower-is-better min(target÷actual,1)×100, no value yet 0% — then Overall = average of those, capped 100. No targets → null.

The representative number on list cards = the latest edition's Post score, always scope-labeled — showing "in progress / —" when that assessment has no recorded values (never a misleading 0). Falls back to the most recent assessment if the latest edition has no Post. No averaging across assessments — ever.

§6 — Comparison · Phase 2

A creation-time wizard, not analytics.

When you build a new assessment / edition, you clone an existing ToC rather than start blank.

Auto-match

What carried over

The wizard auto-matches everything that carried via lineage, and asks you to resolve only the dropped / added diff.

Carry forward

Last year's numbers

Optionally pulls last year's numbers forward — stamped carried_from_prior_edition, never masquerading as a fresh measurement.

One tool, two jobs

Pre→Post & year→year

Within an edition, and across editions — always along a single group thread. It is not the analytics / benchmarks section.

§7 — Removals

The simplification: two concepts, gone.

Both added confusion without truth. Removing them makes the Organization the top of the hierarchy and collapses fieldwork to one honest question.

✕Partners

The implementing partner was a logistical relay — "who forwards the survey link" — not part of the impact story. What we measure is the end beneficiary.

implementing_partnersPartnerManagementPartnerFollowUppartner_fieldwork_status
→ Fieldwork collapses to "of the beneficiaries we're assessing, who responded?"
✕Funder / Grants

No funder entity, no funded-programs relationship, no Grants module. Organization is the top of the hierarchy.

program_fundingFundedProgramCardFundedProgramListFundedProgramComparison
→ A bare budget stays on the program / edition solely to compute Cost-per-Impact — a cost number, not a funder.
§8 — Migration

Guided manual labeling — no auto-backfill.

Existing records aren't uniform, so we don't machine-stamp them. One primitive does the work.

For each existing record, a human assigns it to a (Program, Edition, Assessment) slot. Covers label (new program), merge (two records → one program, different editions/assessments), and split.
Scale

Low-risk

Of 20 active programs, only 4 hold any values; 16 are hollow structure.

Lineage

Fresh keys

Each existing node/indicator gets a fresh unique lineage key — no auto cross-matching (it would manufacture false matches). Historical matches are made on demand via the Phase-2 wizard.

Cleanup

Dropped

Partner & funder data is dropped as part of migration.

§10 — Scope & phasing

Core first, comparison second, connectors later.

Core

Phase 1
  • editions + assessments tables; ToC re-scoped to Assessment; Program Overview page; shared scope-bar + breadcrumb.
  • Create flows: program → auto Edition + Post assessment + empty ToC; "New Edition" (blank or clone); "Add Assessment".
  • Clone-forward + lineage_id stamping.
  • The Link for the survey source — binding, 8 aggregations + custom, Compute & review, provenance, drill-back, stale badge.
  • Per-assessment health score; impact_snapshots re-scoped.
  • Remove Partners; remove Funder/Grants (keep bare budget).
  • Migration tool: (Program, Edition, Assessment) assignment incl. merge/split.
  • Edge-function re-scope; silent-read fixes; routing resolves to the Overview.

Comparison

Phase 2
  • The clone-and-match wizard — auto-match + diff + carry-forward.
  • Year-over-year on the Program Overview.

Later

Future
  • Interview / record / API connectors on the same Link spine.
  • Multi-question index / scale builder.
  • Snapshot automation.
